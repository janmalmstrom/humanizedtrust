Microsoft Security Compliance: Komplett checklista för NIS2-kontroller

Uppdaterad 8 juni 2026 · Lästid: ca 9 minuter · Målgrupp: Ledning & IT-chefer
Sammanfattning
  • NIS2 kräver tio obligatoriska säkerhetskontroller enligt Artikel 21 — många kan implementeras via Microsoft-plattformen.
  • Några kontroller är aktiverade per default i Microsoft 365 och Azure — men ofta inte korrekt konfigurerade.
  • Andra kräver manuell aktivering — och måste dokumenteras för att räknas vid tillsyn.
  • En gap-analys med verktyg som Defender Secure Score och Compliance Manager visar exakt vad som fattas.
  • Denna checklista täcker: MFA (Entra ID), Conditional Access, Defender for Endpoint, Purview DLP, Sentinel-loggning och Intune MDM.
Viktigaste slutsatsen

Microsoft 365 och Azure innehåller alla verktyg för att nå NIS2-efterlevnad — men ingenting är automatiskt. Varje kontroll måste aktiveras, konfigureras och dokumenteras för att räknas. En "default deployment" motsvarar inte NIS2-krav.

Det finns en utbredd missuppfattning: att ett Microsoft 365-abonnemang redan täcker NIS2-kraven. Det gör det inte. Precis som en bil har bromsar från fabrik — men du måste trycka på bromspedalen för att den ska fungera.

Microsoft tillhandahåller verktygen. Din organisation måste aktivera och konfigurera dem enligt NIS2. Och — det viktigaste — du måste dokumentera att det är gjort, så att du kan bevisa det för tillsynsmyndigheten.

Den här artikeln är en praktisk checklista för vad som krävs, vad som redan finns och vad som måste fixas innan nästa granskning.

Varning: En tillsynsmyndighet som granskar din organisation kan begära ut konfigurationsrapporter från dag ett. Om kontrollen inte är dokumenterad räknas den inte — oavsett om verktyget fysiskt finns i systemet.

NIS2:s tio obligatoriska säkerhetskontroller — kort översikt

Innan vi dyker in i Microsoft-plattformen är det viktigt att förstå vad NIS2 faktiskt kräver. Artikel 21 listar tio obligatoriska säkerhetskontroller:

  1. Riskhantering — dokumenterad identifiering, klassificering och bedömning av säkerhetsrisker
  2. Incidenthantering — processer för detektering, analys, rapportering och återhämtning
  3. Affärskontinuitet och återhämtning — backup-, redundans- och disaster recovery-planer
  4. Leverantörssäkerhet — krav på säkerhet i drivarkedjor och tredjeparter
  5. Tillgångskontroll — autentisering, auktorisering och lösenordshantering
  6. Kryptering — skydd av känslig data i vila och under överföring
  7. Övervakning och loggning — centraliserad logginsamling och analys
  8. Skydd mot skadlig kod — antimalware och endpoint protection
  9. Webbtjänstsäkerhet — säkra applikationer och API:er
  10. Fysisk och miljörelaterad säkerhet — åtkomstkontroll till datacenter och serverrum

För organisationer utan egna datacenter (vilket är de flesta) täcks punkt 10 av molnleverantörens SLA och datacenter-certifikationer. De övriga nio måste du dokumentera och implementera.

Sverige har digital suveränitet: Microsoft har investerat 33,7 miljarder SEK i Sweden Central-regionen — ett eget datacentercluster i Gävle, Sandviken och Staffanstorp, driven med 100 % koldioxidfri energi. Det betyder att du kan hålla data inom Sverige och möta både NIS2 och GDPR:s krav på datarezidens samtidigt.

Microsoft 365 och Azure — vad är redan aktiverat?

Om du har Microsoft 365 Business Premium eller Enterprise-prenumerationer innehåller du redan en uppsättning säkerhetskontroller. Men "innehålla" är inte samma som "aktiverat och konfigurerat enligt NIS2".

Här är sammanfattningen av standardläget:

Säkerhetskontroll (NIS2 Artikel 21) Microsoft-produkt Status i default
Riskhantering Defender Secure Score + Compliance Manager Kräver setup
Incidenthantering Defender for Endpoint + Sentinel Kräver aktivering
Affärskontinuitet Microsoft 365 Backup + Azure Backup Delvis aktiverat
Leverantörssäkerhet Entra GDAP + Defender Kräver policy
Tillgångskontroll (MFA, CAP) Entra ID + Conditional Access Kräver konfiguration
Kryptering Purview Sensitivity Labels + Key Vault Kräver aktivering
Övervakning & loggning Sentinel + Defender XDR Kräver aktivering
Skydd mot skadlig kod Defender for Endpoint + Defender for Office 365 Delvis aktiverat
Webbtjänstsäkerhet Defender for API + Cloud App Security Kräver konfiguration
Fysisk säkerhet Azure Datacenter SLA + certifikationer Täckt av leverantör

Som du ser: det mesta är manuell setup. Inget av det här är "klara" kontroller — de är snarare verktyg som du måste sätta ihop till en komplett säkerhetslösning.

Detaljchecklista: Vad måste du göra innan nästa tillsyn?

1. MFA (Multi-Factor Authentication) i Entra ID

NIS2-krav: Artikel 21.5 kräver "starka autentiseringsmekanismer". MFA är det minsta du kan göra.

Status i Microsoft: MFA-funktionaliteten finns i Entra ID (tidigare Azure AD), men det är INTE aktiverat per default för alla användare.

Checklista:

  • ☐ Aktivera MFA för alla användare med administratörsrättigheter (övervägande sannolikhet för angreppsvektor)
  • ☐ Implementera Conditional Access-policy för att kräva MFA baserat på riskfaktorer (inloggningsplats, enhet, tidpunkt)
  • ☐ Aktivera FIDO2-nycklar eller Windows Hello for Business för känsliga roller
  • ☐ Dokumentera MFA-policy i säkerhetshandboken och få styrelsegodkännande
  • ☐ Testa failover för autentisering — vad händer om Microsoft Authenticator-appen inte fungerar?
  • ☐ Lagra recovery codes på ett säkert sätt och testa återhämtning

Tid för aktivering: 2–4 veckor, beroende på organisationsstorlek och hur många externa system som behöver integreras.

2. Conditional Access — Intelligenta åtkomstrestriktioner

NIS2-krav: Artikel 21.5 kräver "lämplig tillgångskontroll baserad på behörighet och behov".

Status i Microsoft: Conditional Access finns som en robust feature i Entra ID Premium — men den är helt svag från start.

Checklista:

  • ☐ Implementera policy för blockering av äldre autentiseringsprotokoll (SMTP, IMAP är höga risker)
  • ☐ Skapa policy för geografisk anomalidetektering — blockera inloggning från omöjliga platser
  • ☐ Konfigurera enhetskrav — endast hanterade enheter (Intune-registrerade) får åtkomst till känslig data
  • ☐ Implementera sessionsbaserad risk — automatisk logout efter inaktivitet, revoke tokens vid hög risk
  • ☐ Testa och felsöka vilka användare som kan logga in från hemmet vs. kontoret
  • ☐ Dokumentera alla Conditional Access-policyer och deras affärslogik i ett konfigurationsregister

Tid för aktivering: 3–6 veckor. Denna är komplex och kräver pilottestning för att inte stänga ute legitima användare.

3. Defender for Endpoint — Endpoint Detection & Response (EDR)

NIS2-krav: Artikel 21.8 kräver "åtgärder för att skydda mot skadlig kod".

Status i Microsoft: Defender for Endpoint är inte inkluderad i standard Microsoft 365 Business Premium. Det är en separat licens (eller inkluderad i vissa E5-paket).

Checklista:

  • ☐ Verifiera att Defender for Endpoint-licensen är tilldelad alla enheter (Windows, Mac, Linux)
  • ☐ Aktivera "Attack Surface Reduction Rules" för att minska angreppsytor
  • ☐ Konfigurera realtidsövervakning och loggning av alla processer och nätverksaktivitet
  • ☐ Integrera med Sentinel för centraliserad loggning och hotanalys
  • ☐ Testa incidentrespons — vad händer när en skadlig fil detekteras?
  • ☐ Implementera automatiserad hotrespons — isolera enheter, blockera användare vid allvarlig risk

Tid för aktivering: 2–3 veckor för deployment, 4–8 veckor för full konfiguration och testning.

4. Purview Data Loss Prevention (DLP) — Kryptering och dataskydd

NIS2-krav: Artikel 21.6 kräver "kryptering av känslig information".

Status i Microsoft: Purview Sensitivity Labels och DLP-policyer finns men är inte aktiverade som standard.

Checklista:

  • ☐ Klassificera all data — känslig, begränsad, intern, offentlig
  • ☐ Skapa Sensitivity Labels för varje dataklass och applicera dem på dokument
  • ☐ Implementera DLP-policyer för att förhindra att känslig data lämnar organisationen (e-post, USB, molntjänster)
  • ☐ Aktivera kryptering av e-post och dokument — använd Purview Encryption
  • ☐ Implementera Azure Key Vault för att hantera krypteringsnycklar
  • ☐ Testa dataförlust-scenarier — vad händer om någon försöker skicka ett klassificerat dokument till Gmail?
  • ☐ Dokumentera varför varje dataklass klassificeras som den gör (för tillsynen)

Tid för aktivering: 4–8 veckor. Denna kräver datamappning först — du måste veta vad du har innan du kan skydda det.

5. Microsoft Sentinel — SIEM och loggning

NIS2-krav: Artikel 21.7 kräver "övervakning och loggning av säkerhetshändelser".

Status i Microsoft: Sentinel är en separat Azure-tjänst. Den är INTE inkluderad i standard Microsoft 365 — du måste köpa den separat.

Checklista:

  • ☐ Etablera en Sentinel-instans i Azure
  • ☐ Anslut alla datakällor: Entra ID, Office 365, Defender for Endpoint, Azure-aktiviteter, firewall-loggar, VPN-loggar
  • ☐ Implementera KQL-frågor (Kusto Query Language) för att detektera misstänkt aktivitet
  • ☐ Skapa alarmer för: brute force-försök, anomal inloggning, datautladdning, konfigurationsförändringar
  • ☐ Etablera en logglagringspolicy — minst 90 dagar aktiva loggar, 1 år archive
  • ☐ Integrera incident-svar — automatisera isolering av enheter, låsning av konton vid allvarlig risk
  • ☐ Testa SOC-teamets responstid på ett test-incident

Tid för aktivering: 6–12 veckor. Det här är en större arkitekturänd och kräver dedikerad tid för query-utveckling och testning.

6. Intune Mobile Device Management (MDM) — Enhetshantering

NIS2-krav: Artikel 21.5 kräver "kontroll över personalnätverk och enheter".

Status i Microsoft: Intune ingår i många Microsoft 365-paket, men det är ofta inte aktiverat eller är konfigurerat för permissiv åtkomst.

Checklista:

  • ☐ Registrera alla företagsägda enheter (Windows, Mac, iOS, Android) i Intune
  • ☐ Skapa device compliance policies — kräv uppdateringar, brandvägg aktiverad, skärmskydd efter X minuter
  • ☐ Implementera App Protection Policies — säker app-åtkomst, låsa äpplet från att kopiera/klistra in mellan appar
  • ☐ Implementera Mobile Application Management (MAM) för BYOD (Bring Your Own Device) om det tillåts
  • ☐ Konfigurera enhet-baserad Conditional Access — saknade patches = blockerad åtkomst
  • ☐ Etablera en process för radering av enheter vid uppsägning (remote wipe)
  • ☐ Testa incidentrespons — kan du fjärrstänga av en komprometterad laptop?

Tid för aktivering: 3–6 veckor, beroende på antalet enheter och heterogenitet (blandning av OS:er).

Implementeringsfall

Scenario: Medelstort konsultföretag, 120 anställda, redan på Microsoft 365 Business Premium.

Situation före: Office 365 och Teams aktiverade. MFA är frivillig. Ingen Conditional Access. Endpoint-skydd saknas. Sentinel inte implementerat. Intune grundläggande.

Åtgärder: Vecka 1–2: Gap-analys med Defender Secure Score. Identifierar 34 automatiska recommendations. Vecka 3–4: Aktivera obligatorisk MFA och basic Conditional Access. Vecka 5–8: Implementera Defender for Endpoint och integrera med Sentinel. Vecka 9–12: Klassificera data, implementera DLP-policyer och Sensitivity Labels. Vecka 13–16: Uppgradera Intune-policyer, testa device compliance.

Resultat: 4 månader senare är organisationen NIS2-ready enligt egen bedömning. Defender Secure Score stiger från 36/100 till 78/100. Styrelsebeslut dokumenterat. Incidenthanteringsplan testerad.

Kostnad: Licensuppgraderingar (Defender, Sentinel): ~150 000 SEK/år. Implementering och konsultering: ~200 000 SEK engångskostnad. Besparing: Undviker potentiella NIS2-böter på 7–10 miljoner euro.

Dokumentation — det viktigaste steget

En kontroll som finns men inte är dokumenterad räknas inte vid tillsyn.

För varje aktiverad kontroll måste du dokumentera:

  • Vad: Vilken kontroll är implementerad? (t.ex. "MFA för alla administratörer")
  • Varför: Vilket NIS2-krav täcker denna kontroll? (t.ex. "Artikel 21.5 — tillgångskontroll")
  • Hur: Vilken Microsoft-produkt eller process? (t.ex. "Entra ID MFA + Conditional Access")
  • När: Datum för aktivering och senaste uppdatering
  • Vem: Ansvarig person eller team
  • Bevis: Screenshots av konfiguration, policy-dokument, testresultat

Samla all denna dokumentation i ett säkert register — Excel, SharePoint, ett confiance management-system. Det är denna dokumentation som du visar tillsynsmyndigheten när de kommer.

Secure Score — ett enkelt framstegsmått

Microsoft Secure Score är ett inbyggt verktyg som bedömer din Azure- och Microsoft 365-miljö på en skala 0–100 poäng.

Det är inte ett perfekt mätetal för NIS2-efterlevnad — men det ger dig ett snabbt överblick över lågt hängande frukt som kan fixas:

  • Aktivera MFA (+1 poäng)
  • Implementera DLP (+2 poäng)
  • Blockera äldre autentisering (+2 poäng)
  • Aktivera Defender for Endpoint (+3 poäng)

NIS2-målnivå: En organisation som måste klara NIS2-granskning bör strevu efter minst 70/100 Secure Score.

Du kan öppna Secure Score i Azure Portal → Security Center → Secure Score. Det finns ingen extra kostnad.
