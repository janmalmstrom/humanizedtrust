SIEM & Loggning

Microsoft Sentinel: Kostnadskalkyl och vad ni faktiskt betalar

Uppdaterad 19 april 2026 · Lästid: ca 8 minuter · Målgrupp: VD, CFO, IT-ledning
Sammanfattning
  • Sentinel prissätts per gigabyte ingesterad data — kostnaderna kan eskalera snabbt om ni inte är medvetna.
  • För en 200–500-personersorganisation ligger rimlig månadskostnad på 15 000–45 000 kronor, beroende på loggvolym.
  • NIS2-krav tvingar er att logga mycket — men inte allt. Kostnadsoptimering betyder smart filtrering, inte skäringning på säkerhet.
  • Alternativa SIEM-lösningar existerar, men Sentinel ger bäst ROI för organisationer redan på Microsoft 365.
Viktigaste slutsatsen

En SIEM-lösning utan kostnadsplanering blir snabbt en dold utgift som överraskar både IT-budget och styrelse. Räkna datavolymen innan du skriver på, inte efter.

Du behöver en SIEM-lösning för att uppfylla NIS2-kraven. Du vet redan detta. Men vad du kanske inte vet är hur snabbt kostnaderna blir en budgetskräck — eller hur du undviker det.

Sentinel är kraftfull, flexibel och integreras naturligt med Microsoft 365. Men prissättningen — per gigabyte ingesterad data — gör att samma lösning kan kosta 10 000 kronor eller 100 000 kronor i månaden beroende på hur väl du förstår vad du ska logga.

Den här artikeln är för dig som vill veta: Vad kostar det? Vad är rimligt för min organisation? Och vad kan jag skippa utan att bryta NIS2-lagen?

Hur fungerar Sentinels prissättning?

Microsoft Sentinel är ett molnbaserat SIEM-verktyg — log aggregation, anomalidetektering, hotresponse i ett gränssnitt.

Priset beräknas så här:

  • Ingestion-kostnad: Du betalar för varje gigabyte data som strömmas in i Sentinel. Ungefär 3,75 SEK per GB (eller motsvarande USD-pris, som varierar efter växelkurs).
  • Retention-kostnad: Hur länge data lagras. Retention mellan 30–90 dagar ingår i ingestion-kostnaden. Längre lagring kostar extra — ungefär 0,55 SEK per GB per månad.
  • Analytic Rules och Automation: De flesta analysregler är gratis. Dyr automation kan räknas separat i vissa fall, men oftast inte.

I praktiken betyder detta: Din månadskostnad beror helt på hur mycket data du för in per dag.

Räknelexempel: En organisation som ingestererar 500 GB data per dag betalar ungefär 500 GB × 3,75 SEK × 30 dagar = 56 250 SEK per månad (enbart ingestion, utan retention). Det är innan någon analytics-regel körs eller någon incident hanteras.

Vad är en rimlig datavolym för 200–500 anställda?

Det finns ingen universell sanning här — det beror på vilka system ni kopplar till Sentinel. Men vi kan skissa ett realistiskt scenario.

För en organisation med 200–500 anställda som kör Microsoft 365, någon on-prem-miljö och ett par företagsapplikationer:

Källsystem Ungefärlig daglig volym Förklaring
Microsoft Defender for Endpoint (EDR) 30–80 GB Process-execution, file-events, network-events från 250–500 endpoints
Microsoft Defender for Office 365 5–15 GB E-post-traffic, phishing-detections, URL-clicks
Azure Active Directory / Entra ID sign-in logs 2–5 GB Login-events, MFA-events, application access
Windows Event Logs (on-prem/hybrid) 10–30 GB Security events, system events från servrar och klienter
Firewall / Network logs 5–20 GB Beroende på vilken brandvägg och konfiguration
Applikationsloggar (verksamhetssystem) 2–10 GB ERP, CRM, databaskörningar — varierar mycket

Totalt: 54–160 GB per dag för en organisation i denna storlek.

Med detta räknat blir månadskostnaden:

  • Låg-end (54 GB/dag): ~20 000 SEK/månad
  • Mitt (100 GB/dag): ~37 500 SEK/månad
  • Högt (160 GB/dag): ~60 000 SEK/månad

Om ni dessutom lagrar data längre än 90 dagar — vilket är rimligt för incidentutredning och compliance — läggs retention-kostnader på. En organisation som lagrar 100 GB/dag under 180 dagar (6 månader) lagring betalar ytterligare ~33 000 SEK/månad för retention.

Varning: Det är lätt att komma upp till 200+ GB per dag om ni börjar logga allt — alla Event ID från Windows, alla syslog-nivåer, alla databastransaktioner. Det kan bli 150 000+ SEK i månaden mycket snabbt utan disciplin.

Vad måste ni logga enligt NIS2?

NIS2 tvingar er att implementera loggning och övervakning. Men texten är inte så specifik att den säger "du måste logga Event ID 4624 från Windows" eller liknande.

Istället säger Artikel 21.2 att ni måste ha:

  • Registrering av säkerhetshändelser (security events)
  • Övervakning av obehörig åtkomst (unauthorized access attempts)
  • Spårning av ändringar i systemkonfiguration (configuration changes)
  • Loggning av administratörsaktiviteter (admin actions)

Det betyder att ni måste logga:

  • Login-events (lyckade och misslyckade)
  • Åtkomst till känslig data
  • Ändringar av användarrättigheter
  • Ändringar av säkerhetskonfiguration
  • Administratörsåtgärder
  • Malware-detections och säkerhetsvarningar
  • Nätverksåtkomst till kritiska system

Vad ni kan skippa utan att bryta NIS2:

  • Varje enskild HTTP-request från webbservrar (logg felkoder, inte alla requests)
  • Alla syslog-nivåer från applikationer (fokusera på ERROR och CRITICAL)
  • Detaljerade databastransaktioner i OLTP-miljöer (om inte data är klassificerad som känslig)
  • Fullständiga nätverkspaketer / deep packet inspection (flow-data räcker)
  • Användaraktivitet i icke-kritiska applikationer
Kostnadsoptimering i praktiken

Scenario: Mediebolag med 280 anställda, på Microsoft 365 + hybrid on-prem servermiljö.

Problem: IT-chefen vill logga allt för "full visibility" och kopplat in alla Windows Event ID:n från servrar och klienter. Ungefärlig datavolym: 210 GB/dag. Kostnad: ~79 000 SEK/månad + retention.

Lösning: Gap-analys identifierar att 70 % av datavolymen kommer från Event ID:n som inte är relevanta för NIS2-krav. IT-chefen justerar loggningen:

  • Bara kritiska Event ID:n från Windows (4624, 4625, 4720, 4728, 4732, 4756) — sparar 50 GB/dag
  • Application logs på ERROR-nivå endast, inte INFO/DEBUG — sparar 30 GB/dag
  • Network flows istället för fullt packet capture — sparar 25 GB/dag
  • Filserver-loggning bara för ändringar av känslig data, inte läsningar — sparar 20 GB/dag

Resultat: Datavolym sjunker till 85 GB/dag. Månadskostnad: ~32 000 SEK istället för 79 000 SEK. Sparar 47 000 SEK per månad. Säkerheten förändras inte — fortfarande fullt NIS2-kompatibel. Styrelsen får grönt ljus för investeringen.

Alternativa SIEM-lösningar — snabbt jämförelser

Sentinel är inte det enda alternativet. Här är hur andra lösningar matchar sig mot prissättning och NIS2-relevans:

SIEM-lösning Prismodell Bäst för NIS2-gap
Splunk Enterprise Årligt avtal, 1 500–5 000 SEK per GB/dag Större organisationer, hybrid-miljöer Mycket bra — etablerad standard
Datadog Security Monitoring Per agent + storage, ~2 000–8 000 SEK/månad för små org Cloud-first och agile-organisationer God, men kräver konfiguration
Elastic Stack (open source) Gratis + hosting ~3 000–15 000 SEK/månad Budget-bewusta organisationer med IT-mognad Bra, men kräver eget arbete
Microsoft Sentinel ~3,75 SEK per GB ingesterad Microsoft 365-organisationer, hybrid Utmärkt — integrerad i Microsoft-stacken
Crowdstrike Falcon LogScale Variabel, ~100–300 USD per TB/dag Endpoint-centric säkerhetsresurser God, fokuserad på threat hunting

För en organisation redan på Microsoft 365 är Sentinel nästan alltid det ekonomiska valet.

Varför? Eftersom Sentinel redan kommer med Microsoft Defender-integrationen utan extra kostnad — du betalar bara för ingestionen. Om du skulle välja Splunk eller Datadog måste du dessutom bygga in samma integreringar själv, vilket kostar konsultarbete.

Om du har en ren on-prem-miljö eller är helt fri från Microsoft kan Splunk eller Elastic Stack vara mindre dyrt. Men då måste du väga det mot behovet av egna integrationer.

Fyra kostnadsbesparingsprinciper

1. Logg inte "allt för säkerhetens skull"

Mer data betyder inte mer säkerhet. Det betyder mer brus, längre analysider och högre kostnader. Logga det som NIS2 tvingar dig att logga — och bara det.

2. Använd filtrering vid källan, inte i Sentinel

Om Windows-servrar skickar logg-data till Sentinel filtreras bäst på servern innan dataöverföring, inte i Sentinel självt. Mindre data in = lägre kostnad.

3. Justera retention baserat på regelverk, inte önsketänkande

NIS2 säger att du ska kunna spåra säkerhetshändelser — men det betyder inte 5 år. De flesta organisationer kan nöja sig med 6–12 månaders retention för incidentutredning. Längre retention för compliance kan ligga i billigare lagring (archive).

4. Automatisera responsen för att minska manuell granskning

Analytics rules i Sentinel kan skapa automationsjobb — t.ex. att låsa ett konto vid N misslyckade inloggningar. Det sparar analystimmar och gör säkerheten snabbare.

Ett ord om dolt dataöverförings-kostnader

Sentinel debiteras för ingesterad data. Men data måste först transporteras från källan — och där kan det gömma sig kostnad.

Om din miljö är helt på Azure / Microsoft 365 är dataöverföringen gratis eller minimal. Men om du har on-prem-servrar eller serverless-funktioner som skickar logg via nätverket kan du stöta på utgående bandbreddskostnader (data egress).

En Log Analytics Agent på en on-prem-server skickar data över internet eller VPN — och detta kan debiteras av din internet-operatör eller din Azure-préstanda-kostnad. Det är vanligtvis små summor (100–500 SEK/månad för små miljöer), men det är värt att veta om.

Budget-planeringschecklista

Innan du godkänner Sentinel-investeringen, gå igenom denna:

  • ☐ Gap-analys genomförd: Vilka system ska loggas? (Se tabell ovan)
  • ☐ Datavolym uppskattad: GB per dag för varje källsystem
  • ☐ Retention-behov definierat: Hur länge måste data lagras?
  • ☐ Compliance-krav kartlagd: Vad säger NIS2, GDPR, andra regler?
  • ☐ Automations-plan: Vilka events behöver automatisk respons?
  • ☐ Monitoring-ansvar klart: Vem övervakar Sentinel dagligen?
  • ☐ Budget för 12 månader: Ingestion + retention + drift
  • ☐ Jämförelse med alternativ: Splunk, Datadog, Elastic — gjort?
