Microsoft Azure vs. AWS: Skysikkerhet for NIS2-etterlevelse

Publisert 19. april 2026 · Lesetid: ca 8 minutter · For: Direktører og styremedlemmer
Sammendrag
  • Både Azure og AWS oppfyller grunnleggende NIS2-krav — valget avhenger av kontekst, ikke plattform alene.
  • Dataresidency er kritisk: NIS2 forventer at sensitive data holder seg innenfor EU/EØS (eller godkjente tredjeland).
  • Delt ansvar er ikke samme som delt risiko — din organisasjon er fortsatt ansvarlig hvis konfigurasjonen er feil.
  • Hendelseshåndtering og sertifiseringer er ikke nok; dokumentasjon av ditt governance-arbeid er obligatorisk.
  • Personlig ansvar for ledelsen gjelder uavhengig av skytilbyder — hvis bakdelen brenner, er det ditt navn i NSM-rapporten.
Viktigste konklusjon

Valg av skyplattform er ikke det som gjør organisasjonen NIS2-klar. Det er dokumentasjon av at du har forstått og godkjent sikkerhetskontrollene — uavhengig av leverandør.

Hvorfor sammenligne Azure og AWS fra NIS2-perspektiv?

De fleste direktører behandler valg av skyplattform som en IT-beslutning. Det er det ikke — det er en NIS2-beslutning.

NIS2-direktivet, som forventes implementert i Norge gjennom EØS-avtalen i løpet av 2025–2026, plasserer personlig ansvar for cybersikkerhet direkte på ledelsens bord. Hvis organisasjonen din bruker Azure eller AWS, og det oppstår en hendelse som kunne vært forhindret gjennom bedre konfigurering eller sikkerhetskontroll, er det ikke skytilbyders skyld alene — det er ditt ledelsesansvar å ha vurdert og godkjent disse valgene på forhånd.

Denne artikkelen sammenligner Azure og AWS ikke som tekniske produkter, men som rammeverk for NIS2-etterlevelse. Vi fokuserer på:

  • Dataresidency og regionale begrensninger
  • Delt ansvarsmodell og sikkerhetskontroller
  • Sertifiseringer og compliance-dokumentasjon
  • Hendelseshåndtering og rapporteringsevne
Advarsel: Begge plattformer støtter NIS2-krav teknisk. Likevel kan en dårlig konfigurert Azure-løsning eller utilstrekkelig AWS IAM-styrne faktisk øke risikoen mer enn en godt styrt on-premises-løsning. Skyplattformen er ikke automatisk sikker — din bruk av den er.

Dataresidency: Hvor må dataene dine bo?

NIS2-direktivet krever at medlemsstater (og EØS-land som Norge) sikrer at kritisk infrastruktur og sensitive data ikke flyttes ut av EU/EØS uten at det foreligger spesifikke rettighetsgrunnlag.

Microsoft Azure: Tilbyr EU/EØS-spesifikke regioner eksplisitt. Du kan velge "Azure Germany" eller "Azure Sweden", og data holder seg fysisk innenfor EØS. Kontrollstrukturen gjør det tydelig hvor dataene skal ligge — og det er juridisk bindende.

Amazon AWS: Tilbyr også europeiske regioner (Frankfurt, Stockholm, Dublin, Paris). Du kan konfigurere slik at data lagres i spesifikke regioner. Men AWS-modellen er mindre streng på regional binding — data kan bevege seg mellom regioner for sikkerhetskopier eller disaster-recovery uten eksplisitt varsel dersom du ikke har konfigurert det selv.

For Norge som EØS-land forventes det at du kan dokumentere at sensitive data holder seg innenfor EØS-grenser. Begge plattformer kan gjøre det, men Azure gjør det mer åpenbart strukturelt.

Sjekkpunkt for direktøren: Spør IT-leder: "Kan vi vise NSM en kart over hvor dataene våre fysisk ligger?" Hvis svaret er "det kommer an på", er det et rødt flagg.

Delt ansvarsmodell — hva betyr det egentlig?

Både Azure og AWS bruker en "delt ansvarsmodell". Dette høres berøygt ut — som om skyleverandøren deler byrden. Det gjør den ikke.

Slik fungerer det:

  • Leverandørens ansvar: Fysisk sikkerhet av datasentre, nettverksinfrastruktur, virtualisering, driftssystemer for servere
  • Ditt ansvar: Identitetsstyring, tilgangskontroll, datakryptering, brannmurer, hendelseslogging, backup-strategi, konfigurasjonsovervåking

Azure har en marginal fordel her: Med Azure Active Directory (nå Entra ID) får du en integrert identitets- og tilgangskontrollplattform som er godt dokumentert for NIS2-sammenheng. MSBs og sektortilsynenes rapporter refererer ofte til Azure som "kjent plattform" — som ikke betyr at det er bedre, men at det er lettere å dokumentere etterlevelse.

AWS krever mer manuell konfigurering av tilsvarende kontroller (IAM, Security Groups, CloudTrail). Det er ikke dårligere — det er bare at du må bevise at du har gjort det, og det tar mer dokumentasjonsarbeid.

Uansett plattform: Når NIS2-tilsyn begynner, skal du kunne vise at du (ikke leverandøren) har konfigurert hver sikkerhetskomponent og kan forklare hvorfor.

Sertifiseringer og compliance-marcos

Begge plattformer er ISO 27001-sertifisert, oppfyller GDPR-krav og har compliance-rapporter som kan hentes ned.

Azure: Tilbyr SOC 2 Type II, ISO 27018 (cloud-spesifikk), og en godt strukturert Compliance Manager som direkte kobler Microsoft 365-løsninger til regelverkkrav (inkludert framtidsperspektivet for NIS2 i Norge).

AWS: Tilbyr SOC 2 Type II, ISO 27001, og AWS Artifact som arkiv for compliance-dokumenter. AWS har også AWS Config, som kan spore endringer i sikkerhetskonfigurasjoner — en viktig funksjon for NIS2.

I praksis: Begge tilbyr tilstrekkelig dokumentasjon for NIS2. Valget kommer ned til hvilken du allerede bruker og hvordan du organiserer governance rundt det.

Område Azure AWS
Dataresidency (EØS) ✓ Eksplisitt regional kontroll ◐ Mulig, krever konfigurering
Identitets- og tilgangskontroll ✓ Entra ID integrasjon ✓ IAM (krever mer setup)
Kryptering (data in transit & rest) ✓ Innebygd + Key Vault ✓ Innebygd + KMS
Hendelseslogging (SIEM-integrasjon) ✓ Azure Monitor + Sentinel ✓ CloudTrail + GuardDuty
ISO 27001 sertifisering ✓ Ja ✓ Ja
Backup & disaster recovery ✓ Azure Backup (immutable vaults) ✓ Diverse løsninger (AWS Backup)
Conformance Manager for NIS2 ✓ Integrert (roadmap) ◐ Manuell kartlegging
Support for hendelsesrapportering til NSM ✓ Dokumentert prosess ◐ Mulig, ikke standardisert

Hendelseshåndtering: Fra oppdagelse til NSM-rapport

NIS2 krever at du skal kunne rapportere alvorlige cybersikkerhetshendelser til NSM innen 24 timer (tidig varsel) og deretter gi en fullstendig rapport innen 72 timer.

Azure: Med Azure Monitor og Microsoft Sentinel (SIEM-løsning) får du sanntidsvarsel om anomalier, integrasjon med inntrengingsdeteksjon og automatiserte respons-playbooks. Sentinel kan konfigureres til å sende varslinger direkte til SOC-teamet ditt og kan bidra til dokumentasjon av den kritiske 24-timers-fristen.

AWS: CloudTrail loggfører alle API-kall, og GuardDuty tilbyr trusselsdeteksjon. Du må integrere disse manuelt (eller bruke tredjeparts-SIEM). AWS tilbyr ikke samme integrerte hendelseshåndteringskjede som Azure — du bygger det selv.

Begge kan klare NIS2-kravene. Azure er bare enklere å konfigurere for etterlevelse — og når NSM ringer, er det en fordel.

Sak: Hendelserappering under press

En norsk e-handelsbedrift (viktig entitet under NIS2) bruker AWS. En ransomware-operatør krypterer kritisk data 15. februar 2026. IT-leder oppdager det 16. februar klokken 09:00.

Scenario A (dårlig forberedt): IT-leder ringer direktør og CFO. De diskuterer om de skal rapportere. CloudTrail-logger må granskes manuelt. 24-timers-fristen glir og glir. Ved klokken 18:00 på 17. februar (39 timer etter oppdagelse) sendes endelig varsel til NSM. Straffrunde 1 er tapt — for sent varsel.

Scenario B (forberedt): Samme bedrift, samme angrep. Men de bruker Azure med Sentinel. Anomalier ble detektert automatisk klokken 10:30. En respons-playbook validerete alvorlighetsgraden. NSM ble varslet klokken 11:45 samme dag (2 timer etter oppdagelse). 72-timersrapporten ble levert klokken 10:00 på 19. februar med full loggdokumentation.

Utfall: Scenario B oppfyller NIS2-fristen. Scenario A risikerer straffrunde for brudd på rapporteringsplikt (i tillegg til hendelsen selv). Rollen til skyplattformen var ikke avgjørende — det var organisasjonens forberedelse.

Leverandøravhengighet og låsingsrisiko

En ofte oversett risiko i skymigrering: Leverandøravhengighet.

Azure: Løsninger bygget rundt Microsoft-økosystemet (Microsoft 365, Teams, Dynamics) blir tett integrert. Det gjør det vanskelig (dyrere og mer arbeidskrevende) å migrere senere. Fra NIS2-perspektiv betyr dette at du er låst til Microsofts siktlinje for regelverksetterlevelse — hvis Microsoft ikke prioriterer norsk NIS2-implementering, er du avhengig av dem.

AWS: Mer åpen arkitektur gjør det lettere å bytte komponenter eller migrere senere. Fra NIS2-perspektiv er det mindre risiko for leverandør-diktert etterlevelse, men større risiko for at etterlevelsen krever mer arbeid fra ditt hold.

Dette er ikke et teknisk spørsmål — det er et governance-spørsmål. Begge er akseptable under NIS2, så lenge du kan dokumentere hvorfor du valgte som du gjorde.
